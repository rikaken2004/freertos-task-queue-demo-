# 基于FreeRTOS的智能分拣模拟系统

## 项目简介

这个项目是我学STM32和FreeRTOS过程中的一个练习，用STM32F103C8T6做了一个模拟的智能分拣系统。核心想法很简单：三个独立的任务同时跑，互相之间通过FreeRTOS的队列来通信，模拟一个货物扫描、分拣、系统监控的流程。

我是跟着教程学完了GPIO、外部中断、定时器这些外设之后，感觉裸机写的代码越来越难维护，才决定试试FreeRTOS。这个项目就是用来检验自己到底有没有搞懂多任务调度和队列通信的。（根据上一个51单片机项目简单更改而来）

## 硬件连接

| 外设 | 连接引脚 | 说明 |
|------|----------|------|
| 按键 | PB1 | 模拟光电传感器，按一次就当扫到一个货 |
| 绿灯 | PA15 | 模拟传送带的电机，快闪3下表示货被送走了 |
| 红灯 | PB12 | 系统心跳灯，1秒闪一下，告诉我系统还活着 |

实物是在面包板上搭的，用杜邦线连接STM32最小系统板和LED、按键。接线图在Docs文件夹里。

## 软件架构

整个代码分两大块：

`main.c` 只干一件事：初始化硬件，然后启动FreeRTOS调度器。那个`while(1)`大循环在我这个工程里永远跑不到，因为我用了`vTaskStartScheduler()`，这行代码一执行，CPU就交给FreeRTOS管了，下面的死循环根本没机会跑。

`freertos.c` 才是真正的业务代码。里面有一个`MX_FREERTOS_Init()`函数，负责创建队列和三个任务。三个任务函数各自有一个自己的`for(;;)`死循环，互不干扰。

## 三个任务都在干什么

**传感器任务（优先级3）**：每10ms扫描一次按键，做了20ms的消抖，等按键松手了才算一次有效触发。按满5次就往队列里塞一条消息，然后计数器清零，准备下一轮。这个任务的延时用的是`vTaskDelay`，不是裸机常用的`Delay_ms`，因为用FreeRTOS的延时可以让任务进入阻塞态，CPU会自动去跑别的任务。

**电机任务（优先级3）**：一直在`xQueueReceive`那个地方死等。队列里没消息的时候，这个任务就完全休眠了，不占任何CPU资源。一旦传感器任务往队列里发了消息，FreeRTOS内部会自动把这个任务唤醒，然后绿灯闪3下。闪完又回到`xQueueReceive`继续等。

**心跳任务（优先级1）**：优先级最低，只干一件事——每1秒翻转一次红灯。这是一个最典型的"后台任务"，只在另外两个任务都在休息的时候才跑一下，告诉我系统没死。

## 为什么要用FreeRTOS原生API

CubeMX自动生成代码的时候，会默认用CMSIS-RTOS V2那一套封装，比如`osThreadNew`、`osMessageQueueNew`。我折腾了一圈，最后选了直接用FreeRTOS原生API（`xTaskCreate`、`xQueueCreate`这些）。

原因有两点：第一，用原生API我写的代码拿到别的芯片上也能直接用，不依赖ST公司的封装层。第二，调试的时候少一层中间层，出了问题更容易定位到是FreeRTOS内核的行为还是封装层的问题。

## 为什么用队列而不是全局变量

我之前学裸机的时候，通信基本都是靠全局变量加轮询。但在这个项目里，如果我用全局变量，电机任务就得不停检查变量值，CPU一直在空转。换成队列之后，电机任务可以在队列为空时直接休眠，有消息才被唤醒。而且队列内部做了数据拷贝，不会因为传感器任务写数据和电机任务读数据之间时序冲突导致数据被覆盖。

## 裸机开发和FreeRTOS的区别（我自己的体会）

| | 裸机`while(1)`大循环 | FreeRTOS多任务 |
|------|------|------|
| 代码结构 | 所有逻辑串行排在一个循环里 | 每个任务各自一个`for(;;)` |
| 延时等待 | CPU死等 | 任务休眠，CPU自动切走 |
| 通信方式 | 全局变量加轮询检查 | 队列加事件驱动 |
| 加新功能 | 要调整整个循环的时间分配 | 新建一个任务就行了 |

## 怎么用逻辑分析仪验证

我做这个项目的时候，发现万用表的表笔太粗，根本塞不进面包板的孔。后来买了个逻辑分析仪，这个问题才解决。逻辑分析仪的杜邦线可以直接插在面包板上，完美替代万用表的电压测量功能。

验证的时候，我把CH1接PA15（绿灯），CH2接PB12（红灯），采样时间设2秒以上，就能同时看到两路波形。红灯每秒翻转一次，按5次按键后绿灯快闪3下。两个波形互不影响，证明三个任务是真的在并发跑，不是裸机那种顺序执行。波形截图在Docs文件夹里。
